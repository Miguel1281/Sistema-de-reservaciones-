| Campo | Descripción

| ID
| CU-01

| Nombre
| Hacer RESERVA

| Autor
| Brayan Gabriel Fernández Tlapa

| Descripción
| Permite al HUESPED asegurar el arrendamiento de una HABITACION para un periodo específico. Este proceso es el núcleo del negocio, ya que garantiza la ocupación del inventario y la generación de ingresos directos, facilitando la conversión de un visitante web en un cliente confirmado.

| Actor(es)
| HUESPED

| Disparador
| El HUESPED selecciona la opción para reservar.

| Precondiciones
| PRE-01: Existe un HOTEL seleccionado. +
PRE-02: Existe una HABITACION seleccionada.

| Flujo Normal
| 1. El sistema muestra un formulario para la captura de datos del HUESPED (nombre, apellidos, correo Electronico, telefono). +
2. El HUESPED (nombre, apellidos, correo Electronico, telefono) ingresa su información personal confirma la operación. (Ver FA-2a) +
3. El sistema valida los datos ingresados y muestra un resumen de todas las selecciones realizadas anteriormente. (Ver FA-3a) (Ver FA-3b) (Ver EX-01) +
4. El HUESPED selecciona la opción para proceder al pago. (Ver FA-2a) +
5. El sistema retiene temporalmente la HABITACION del inventario, crea la RESERVA con un estado de "Pendiente". (Ver EX-1) (Ver EX-02) +
6. El sistema comienza con el proceso de cobro (incluye CU-02 Realizar pago de reserva). +
7. El Sistema recibe la confirmación exitosa del pago cambia el estado de la RESERVA a "Confirmada" y muestra el mensaje de confirmación al HUESPED. (Ver EX-02) +
8. El HUESPED selecciona aceptar. +
9. Termina el caso de uso

| Flujos Alternos
| *FA-2a: El HUESPED cancela la operación.* +
1. El HUESPED selecciona la opción de cancelar. +
2. El sistema cancela la operación. +
3. Termina el CU. +
*FA-3a: Se ingresaron datos incompletos* +
1. El sistema muestra un aviso para informar que no se ingresaron todos los datos. +
2. El HUESPED completa los datos faltantes. +
3. Regresa al paso tres del flujo normal. +
*FA-3b: Se ingresaron datos inválidos* +
1. El sistema muestra un aviso para informar que se ingresaron datos inválidos. +
2. El HUESPED corrige los datos. +
3. Sigue al paso tres del flujo normal.

| Excepciones
| *EX-01: Mientras el HUESPED llenaba sus datos, otro usuario completó la reserva de la última HABITACION disponible de ese tipo.* +
1. El sistema muestra un mensaje de aviso de que la HABITACION ya no está disponible. +
2. El HUESPED selecciona la opcion para aceptar. +
3. Se regresa al paso uno. +
*EX-02: No hay conexión con la base de datos.* +
1. El sistema muestra un aviso de que no hay conexión. +
2. El HUESPED selecciona la opción aceptar. +
3. Termina el caso de uso.

| Postcondiciones
| POST-01: Existe una reserva con el estado "Confirmada" asociada a un HUESPED.

| Reglas de negocio
| RN-01: El precio total mostrado en el resumen debe ser exactamente el monto enviado a la pasarela de PAGO.

| Incluye
| Incluye al CU-02 Realizar pago de reserva

| Campo | Descripción

| ID
| CU-02

| Nombre
| Realizar PAGO de RESERVA

| Autor
| Brayan Gabriel Fernández Tlapa

| Descripción
| Permite al HUESPED liquidar el importe total de la ESTANCIA asociada a su solicitud. El sistema debe coordinarse con una entidad financiera externa para validar los fondos y asegurar la transacción, cumpliendo con la regla de negocio de "pago único" para confirmar definitivamente la reserva.

| Actor(es)
| HUESPED

| Disparador
| EI HUESPED decide proceder al pago de una reserva que se encuentra en estado "Pendiente de Pago".

| Precondiciones
| PRE-01: Existe una RESERVACION creada y vigente asociada al HUESPED.

| Flujo Normal
| 1. El Sistema recupera y presenta el Resumen de Costos de la RESERVACION (monto total, concepto, fechaLimite para completar la operación). +
2. EI HUESPED ingresa la Información Financiera requerida (número Tarjeta, nombre Titular, fecha Vencimiento, código Seguridad) y confirma la orden de PAGO. +
3. El sistema valida los datos ingresados. (Ver FA-3a) (Ver FA-3b) +
4. El Sistema verifica que no exista una transacción exitosa previa para esta misma RESERVA. +
5. Muestra una página de confirmación con el número de RESERVA y envía un correo electrónico con el resumen de la RESERVA. +
6. EI HUESPED selecciona aceptar. +
7. Termina el caso de uso.

| Flujos Alternos
| *FA-2a: EI HUESPED decide cancelar el pago y hace clic en cancelar.* +
1. El sistema libera la HABITACION del inventario. +
2. La RESERVA se marca como "Cancelada". +
3. El sistema regresa al HUESPED a la página principal. +
4. Termina el caso de uso. +
*FA-3a: Se ingresaron datos incompletos* +
1. El sistema muestra un mensaje de para informar que no se ingresaron todos los datos. +
2. EI HUESPED completa todos los datos. +
3. Sigue al paso uno del flujo normal. +
*FA-3b: Se ingresaron datos inválidos* +
1. El sistema muestra un mensaje de para informar que se ingresaron datos inválidos. +
2. EI HUESPED completa todos los datos. +
3. Sigue al paso uno del flujo normal.

| Excepciones
| *EX-01: La pasarela de pagos rechaza la transacción (fondos insuficientes, datos incorrectos).* +
1. El sistema muestra una aviso al CLIENTE de que se rechazó la transacción. +
2. Regresa el paso dos del flujo normal. +
*EX-02: No hay conexión con la base de datos.* +
1. El sistema muestra un aviso de que no hay conexión. +
2. EI HUESPED selecciona la opción aceptar. +
3. Termina el caso de uso.

| Postcondiciones
| POST-01: Existe un PAGO exitoso para una RESERVA.

| Reglas de negocio
| RN-01: El sistema debe garantizar que cada intento de cobro sea único para evitar dobles cargos.

2. EI HUESPED selecciona aceptar +

*FA-4a: EI HUESPED cancela la selección de una habitación.* +
1. EI HUESPED selecciona la opción de cancelar. +
2. Se regresa al paso uno del flujo normal.

| Excepciones
| *EX-01: No hay conexión con la base de datos.* +
1. El sistema muestra un aviso de que no hay conexión. +
2. EI HUESPED selecciona la opción aceptar. +
3. Termina el caso de uso

| Campo | Descripción

| ID
| CU-04

| Nombre
| Cancelar RESERVA

| Autor
| Brayan Gabriel Fernández Tlapa

| Descripción
| Permite al HUÉSPED rescindir el contrato de arrendamiento temporal de una HABITACION previamente confirmada. Este proceso es fundamental para la organización ya que permite recuperar la disponibilidad del inventario para su reventa inmediata, maximizando la probabilidad de ocupación, y gestiona la devolución de recursos económicos al HUESPED conforme a las normativas vigentes, asegurando la transparencia financiera.

| Actor(es)
| HUESPED

| Disparador
| EI HUÉSPED selecciona la opción para cancelar una RESERVA.

| Precondiciones
| PRE-01: Existe una RESERVA asociada al HUÉSPED con estado "Confirmada". +
PRE-02: La fecha actual es anterior a la fecha de check-in registrada en la reserva.

| Flujo Normal
| 1. El sistema valida la fecha actual contra la fecha de llegada y las políticas de cancelación, calculando el monto a reembolsar o la penalización aplicable. (Ver EX-01) (Ver FA-1a) +
2. El sistema muestra un resumen con las consecuencias de la cancelación (monto a devolver, cargos por penalización) y solicita confirmación definitiva. (Ver EX-01) +
3. EI HUÉSPED confirma la acción de cancelar la reserva. (Ver FA-3a) +
4. El sistema procesa la devolución de fondos a través de la pasarela de pagos si aplica reembolso. (Ver EX-02) (Ver EX-01) +
5. El sistema actualiza el estado de la RESERVA a "Cancelada" y libera inmediatamente la HABITACIÓN en el inventario. (Ver EX-01) +
6. El sistema muestra un mensaje de éxito indicando que la RESERVA ha sido cancelada y envía un comprobante al correo del HUÉSPED. +
7. EI HUESPED selecciona la opción de aceptar. +
8. Termina el caso de uso

| Flujos Alternos
| *FA-1a: La cancelación está fuera de política y no cumple con las políticas para un reembolso.* +
1. El sistema informa al HUESPED que la cancelación no es posible. +
2. EI HUESPED da clic en aceptar +
3. Regresa al paso uno del flujo normal +
*FA-3a: EI HUESPED no confirma la cancelación.* +
1. EI HUESPED selecciona la opción de cancelar +
2. Termina el caso de uso

| Excepciones
| *EX-01: No hay conexión con la base de datos.* +
1. El sistema muestra un aviso de que no hay conexión. +
2. EI HUESPED selecciona la opción aceptar. +
3. Termina el caso de uso +
*EX-02: Fallo en la transacción de reembolso.* +
1. La pasarela de pagos rechaza la operación de devolución. +
2. El sistema marca la RESERVA con un estado intermedio "Cancelada - Reembolso Pendiente". +
3. El sistema libera la HABITACIÓN (priorizando el inventario según los objetivos de negocio). +
4. El sistema muestra un mensaje informando del problema. +
5. EI HUESPED selecciona confirmar +
6. Termina el caso de uso.

| Postcondiciones
| POST-01: El estado de la RESERVA es "Cancelada". +
POST-02: La HABITACIÓN asociada está marcada como disponible en el inventario global para las fechas correspondientes.

| Reglas de negocio
| RN-01: La elegibilidad para un reembolso depende de las políticas vigentes de la cadena de hoteles.

| ID
| CU-06

| Nombre
| Modificar periodo de ESTANCIA

| Autor
| Brayan Gabriel Fernández Tlapa

| Descripción
| Permite al HUÉSPED ajustar las fechas de entrada o salida de una RESERVA confirmada. Este proceso es clave para ofrecer flexibilidad comercial al cliente y optimizar la ocupación del inventario, asegurando que cualquier cambio en la duración del servicio se refleje inmediatamente en la disponibilidad de habitaciones y en el balance financiero (cobro de diferencias o aplicación de reembolsos).

| Actor(es)
| HUESPED

| Disparador
| EI HUÉSPED selecciona la opción "Modificar Fechas" dentro del detalle de una RESERVA activa.

| Precondiciones
| PRE-01: La RESERVA se encuentra en estado "Confirmada". +
PRE-02: La solicitud se realiza antes de la fecha de check-in original (o dentro de los límites permitidos por la política de modificación).

| Flujo Normal
| 1. El sistema recupera los detalles de la RESERVA actual y presenta un selector de fechas pre-llenado. (Ver EX-01) +
2. EI HUÉSPED ingresa el nuevo rango de fechas deseado. +
3. El sistema valida la disponibilidad del inventario para las nuevas fechas y verifica las políticas de estancia mínima/máxima. (Ver FA-3a) (Ver FA-3b) +
4. El sistema calcula el nuevo costo total de la estancia y determina la diferencia financiera respecto al pago original. +
5. El sistema muestra el resumen de la modificación si se debe reembolsar o cobrar. +
6. EI HUÉSPED confirma la modificación y acepta los nuevos costos. (Ver FA-6a) +
7. El sistema gestiona la transacción financiera (reembolso cobro). +
8. El sistema actualiza las fechas de la RESERVA y ajusta el bloqueo de inventario (libera días o reserva nuevos). (Ver EX-01) +
9. El sistema muestra y envía la confirmación actualizada al correo del HUÉSPED. +
10. EI HUESPED selecciona la opción de confirmar. +
11. Termina el caso de uso

| Flujos Alternos
| *FA-3a: No hay disponibilidad para las nuevas fechas.* +
1. El sistema informa al HUESPED que no hay disponibilidad y le sugiere intentar con otro rango de fechas. +
2. EI HUESPED selecciona la opcion de confirmar. +
3. El flujo regresa al paso 2. +
*FA-3b: El cambio no cumple con las políticas.* +
1. El sistema informa al HUESPED que su solicitud no es permitida por las políticas vigentes. +
2. Termina el caso de uso. +
*FA-6a: EI HUÉSPED rechaza la cotización (Costos muy altos).* +
1. EI HUÉSPED selecciona la opción para cancelar modificacion. +
2. El sistema descarta los cálculos temporales. +
3. Termina el caso de uso.

| Excepciones
| *EX-1: No hay conexión con la base de datos.* +
1. El sistema muestra un aviso de que no hay conexión. +
2. EI HUESPED selecciona la opción aceptar. +
3. Termina el caso de uso

| Postcondiciones
| POST-01: La RESERVA refleja el nuevo rango de fechas de manera persistente. +
POST-02: El inventario se ha actualizado (se han ocupado nuevos días y/o liberado días anteriores).

| Reglas de negocio
| RN-01: Las modificaciones están sujetas a las políticas de cambio de la cadena de hoteles.

| ID
| CU-07

| Nombre
| Modificar tipo de HABITACION

| Autor
| Brayan Gabriel Fernández Tlapa

| Descripción
| Permite al HUÉSPED cambiar la categoría de la HABITACION asignada a su RESERVA. Este proceso busca maximizar los ingresos mediante la venta de mejoras (upselling) o acomodar las necesidades cambiantes del cliente, asegurando que el inventario se actualice correctamente sin duplicar RESERVA(s) ni perder la asignación original hasta que el cambio sea definitivo.

| Actor(es)
| HUESPED

| Disparador
| EI HUÉSPED selecciona la opción "Cambiar categoría de habitación" dentro de los detalles de una RESERVA.

| Precondiciones
| PRE-01: EI HUESPED tiene una RESERVA confirmada. +
PRE-02: Existen otros tipos de habitación definidos en el hotel.

| Flujo Normal
| 1. El sistema consulta el inventario disponible para las fechas de la reserva y recupera los TIPOSDEHABITACION alternativos con sus tarifas actuales. (Ver EX-01) +
2. El sistema verifica si existen opciones disponibles diferentes a la actual. (Ver FA-2a) +
3. El sistema muestra una lista comparativa con las opciones disponibles +
4. EI HUESPED selecciona el nuevo tipo de HABITACION. (Ver FA-2a) +
5. El sistema retiene temporalmente una unidad del nuevo tipo de habitación (sin liberar la anterior aún). (Ver FA-5a) +
6. El sistema presenta el resumen financiero final y solicita confirmación. +
7. EI HUESPED acepta los cambios. (Ver FA-4a) +
8. El sistema procesa la transacción financiera (Cobro de diferencia o Reembolso). (Ver EX-02) +
9. El sistema libera la HABITACION original al inventario general y consolida la nueva HABITACION en la RESERVA. (Ver EX-01) +
10. El sistema actualiza el registro de la RESERVA y envía la notificación de cambio exitoso. (Ver EX-01) +
11. Termina el caso de uso

| Flujos Alternos
| *FA-4a: EI HUESPED cancela la operacion* +
1. EI HUESPED selecciona la opcion para cancelar. +
2. Regresa al paso anterior del flujo normal. +
*FA-2a: No hay otros tipos de HABITACION disponibles.* +
1. El sistema informa al HUESPED que no hay otras opciones de HABITACION para sus fechas. +
2. EI HUESPED selecciona la opcion de aceptar. +
3. Termina el caso de uso +
*FA-5a: Otro usuario tomó la última habitación disponible.* +
1. El sistema informa que la HABITACION seleccionada ya no está disponible +
2. EI HUESPED selecciona la opcion de confirmar. +
3. Regresa al paso 3.

| Excepciones
| *EX-01: No hay conexión con la base de datos.* +
1. El sistema muestra un aviso de que no hay conexión. +
2. EI HUESPED selecciona la opción aceptar. +
3. Termina el caso de uso +
*EX-02: La pasarela rechaza el cobro adicional.* +
1. El sistema libera inmediatamente la retención de la nueva HABITACION. +
2. El sistema mantiene la RESERVA con la HABITACION original. +
3. El sistema informa del error de pago al HUÉSPED. +
4. EI HUESPED selecciona la opcion aceptar +
5. Termina el caso de uso.

| Postcondiciones
| POST-01: La RESERVA está asociada al nuevo TIPOHABITACION

| Reglas de negocio
| RN-01: Al seleccionar una nueva categoría, el sistema debe verificar disponibilidad considerando el overbooking permitido, pero no debe liberar la HABITACION original hasta que el PAGO de la diferencia (si aplica) sea exitoso. +
RN-02: El cobro o reembolso se basa en la diferencia de tarifa vigente al momento del cambio, no al momento de la RESERVA original.

=== Recepcionista

| Campo | Descripción

| ID
| CU-08

| Nombre
| Check in

| Autor
| Brayan Gabriel Fernández Tlapa

| Descripción
| Permite al RECEPCIONISTA formalizar el inicio de la ESTANCIA del HUÉSPED en el HOTEL. Su propósito es validar la identidad del titular, confirmar que la RESERVA está pagada y vigente, y entregar el acceso a la HABITACION (físico o digital), cambiando el estado legal y operativo de la unidad a "Ocupado" para fines de seguridad y control de inventario.

| Actor(es)
| RECEPCIONISTA

| Disparador
| El RECEPCIONISTA selecciona la opción "Registrar Llegada" o busca una reserva activa al arribo del HUÉSPED.

| Precondiciones
| PRE-01: Existe una RESERVA confirmada con fecha de entrada igual a la fecha actual.

| Flujo Normal
| 1. El sistema presenta el buscador de reservas del día. (Ver EX-01) +
2. El RECEPCIONISTA ingresa los datos del HUÉSPED(nombre) y ejecuta la búsqueda. +
3. El sistema recupera la reserva y muestra el estado del pago y la habitación asignada. (Ver FA-3a) (Ver EX-01) +
4. El RECEPCIONISTA confirma la operación. +
5. El sistema cambia el estado de la HABITACIÓN de "Reservada/Disponible" a "Ocupado". (Ver EX-01) +
6. El sistema registra la hora exacta de entrada. (Ver EX-01) +
7. Termina el caso de uso

| Flujos Alternos
| *FA-3a: La RESERVA no es encontrada.* +
1. El sistema informa que no encuentra ninguna RESERVA activa. +
2. El RECEPCIONISTA selecciona aceptar. +
3. Termina el caso de uso

| Excepciones
| *EX-01:No hay conexión con la base de datos.* +
1. El sistema muestra un aviso de que no hay conexión. +
2. El RECEPCIONISTA selecciona la opción aceptar. +
3. Termina el caso de uso

| Postcondiciones
